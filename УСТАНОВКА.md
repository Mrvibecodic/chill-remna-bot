# Установка бота Remnawave (chill-remna-bot)

Почти вся настройка — прямо в Telegram (мастер сам задаёт вопросы и всё проверяет).
На сервере вы только клонируете репозиторий, вписываете в `docker-compose.yml` два
токена и запускаете. Файл уже настроен на самоменеджмент: бот стартует **без базы
данных** и сам поднимет нужную БД по вашему выбору, а также умеет обновляться.

Репозиторий: https://github.com/Mrvibecodic/chill-remna-bot

> Реализована установка бота (выбор БД + подключение панели). Продажи и платёжки — на следующих фазах.

---

## 1. Предусловия

1. Сервер с Linux, установлены **Docker** и **Docker Compose v2** (`docker compose version`).
2. Уже работающая панель **Remnawave**.
3. **Токен бота** от [@BotFather](https://t.me/BotFather) (`/newbot`).
4. **Ваш Telegram ID** (число) от [@userinfobot](https://t.me/userinfobot).
5. **API-token панели** (создаётся в дашборде, роль API) — введёте в мастере.

---

## 2. Деплой (готовый образ из GHCR)

Образ собирает CI и публикует в GitHub Container Registry: `ghcr.io/mrvibecodic/chill-remna-bot:latest`. На сервере **сборки нет** — берём `docker-compose.yml`, вписываем 2 значения и тянем готовый образ.

**Шаг 1. Папка и docker-compose.yml:**
```bash
mkdir -p /opt/remnachillbot && cd /opt/remnachillbot
curl -fsSL -o docker-compose.yml \
  https://raw.githubusercontent.com/Mrvibecodic/chill-remna-bot/main/docker-compose.yml
```
> Пока репозиторий/образ приватные: получить файл клоном (`git clone … .`) и один раз
> `echo ВАШ_PAT | docker login ghcr.io -u Mrvibecodic --password-stdin` (PAT с `read:packages`).
> После перевода в публичные — не нужно.

**Шаг 2. Впишите токены** — в `docker-compose.yml` две строки:
```bash
nano docker-compose.yml
```
```yaml
      BOT_TOKEN: "123456789:AAE..."          # от @BotFather
      ADMIN_TELEGRAM_ID: "000000000"          # число, от @userinfobot
```
Остальное менять не нужно. Если ставите НЕ в `/opt/remnachillbot` — поправьте путь
`/opt/remnachillbot:/compose` и `COMPOSE_HOST_DIR`.

**Шаг 3. Скачивание образа и запуск:**
```bash
docker compose pull
docker compose up -d
docker compose logs -f
```
Образ тянется готовым из GHCR (быстро, без нагрузки на сервер). Сети вручную создавать не нужно.

**Шаг 4.** В Telegram боту — **/start** (мастер, см. ниже). Панель на другом сервере — выбирайте «Удалённо».

**Обновление:** `docker compose pull && docker compose up -d` (либо команда `/update` в боте — скачает свежий образ и перезапустит).

## 3. Мастер установки в Telegram

Каждый шаг — вопрос с пояснением, вы жмёте кнопку или присылаете значение.

1. **Язык** — RU / EN.
2. **База данных:**
   - **SQLite** — бот создаёт файл в томе `bot-data`, ничего больше не поднимается.
   - **PostgreSQL 17** — бот сам допишет сервис `db` в `docker-compose.yml`, поднимет его
     (через docker.sock) и переключится на него без перезапуска. Если до этого были
     данные в SQLite — они переносятся. Если поднять не удалось — бот спокойно остаётся
     на SQLite.
3. **Где бот относительно панели:** Локально (одна docker-сеть, рекомендуем) / Удалённо.
4. **Как установлена панель** (только для «Удалённо»): Официально (Caddy) / Скрипт eGames (nginx).
5. **Реквизиты:**
   - URL панели (для «Удалённо»); для «Локально» подставляется `http://remnawave:3000`.
   - API-token панели.
   - **Кука** — только «Удалённо» + eGames. Бот подскажет: откройте на сервере панели
     `/opt/remnawave/nginx.conf`, блок `map $http_cookie $auth_cookie { "~*ИМЯ=ЗНАЧЕНИЕ" 1; }`.
     ИМЯ и ЗНАЧЕНИЕ — два случайных набора из 8 латинских букв. Пришлите `ИМЯ=ЗНАЧЕНИЕ`.
   - **X-API-Key** — только «Удалённо» + Caddy, и только если `/api` защищён.
6. **Проверка связи** — бот сам дёргает `/api/system/health` и показывает число пользователей.

---

## 4. Команды

- **/status** — связь с панелью + краткая статистика.
- **/update** — скачивание свежего образа из GHCR и перезапуск (`docker compose pull` + `up -d`, через одноразовый контейнер-контроллер).
- **/setup** — пройти мастер заново.

---

## 5. Если что-то не так

- **Локально «нет связи с панелью»** → бот сам подключается к сети панели; убедитесь, что её контейнер запущен и называется `remnawave` (иначе задайте `PANEL_CONTAINER`).
- **Локально «нет связи с панелью»** → контейнер панели должен называться `remnawave`
  и быть в сети `remnawave-network` (`docker network inspect remnawave-network`).
- **HTTP 401/403** → неверный/просроченный API-token, создайте новый и /setup.
- **Удалённо + eGames, соединение рвётся (444)** → неверная кука, перепроверьте `ИМЯ=ЗНАЧЕНИЕ`.
- **PostgreSQL не поднялся** → бот остаётся на SQLite; проверьте, что проброшен docker.sock
  (он есть в `docker-compose.yml`), и повторите выбор позже.

Логи: `docker compose logs -f` (из каталога `/opt/remnachillbot`).

---

## 6. Обновление и бэкап

**Обновление:** команда `/update` боту, либо вручную:
```bash
cd /opt/remnachillbot && docker compose pull && docker compose up -d
```

**Бэкап:**
- SQLite — том `bot-data` (файл `bot.db` + `secret.key`). Не теряйте `secret.key` —
  без него не прочитать зашифрованные настройки.
- PostgreSQL — `pg_dump` базы `remnabot` из контейнера `remnabot-db`, плюс том `bot-data` ради `secret.key`.

---

## 7. Что задаётся где

| Параметр | Где |
|---|---|
| Токен бота, Telegram ID | строки в `docker-compose.yml` |
| Движок БД, URL панели, API-token, кука, X-API-Key | мастер в Telegram |
| Язык, режим, способ установки панели | мастер в Telegram |

Всё, что введено в мастере, хранится в БД зашифрованным и переживает перезапуск —
повторно проходить мастер не нужно.
